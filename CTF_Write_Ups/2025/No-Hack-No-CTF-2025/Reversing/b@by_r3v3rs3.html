> By. LemonTea

I have been given the following file type :

file challenge.wasm    
challenge.wasm: WebAssembly (wasm) binary module version 0x1 (MVP)

This is a Web Assembly file and can be disassembled. I can use

wasm2wat --help  
usage: wasm2wat [options] filename  
  
 Read a file in the WebAssembly binary format, and convert it to  
 the WebAssembly text format.  
  
examples:  
 # parse binary file test.wasm and write text file test.wast  
 $ wasm2wat test.wasm -o test.wat  
  
 # parse test.wasm, write test.wat, but ignore the debug names, if any  
 $ wasm2wat test.wasm --no-debug-names -o test.wat

And it gives:

wasm2wat challenge.wasm -o challenge.wat

file challenge.wat    
challenge.wat: ASCII text, with very long lines (4067)

This file can be opened in a text editor and contains a total of 5877 lines at the end we can see some interesting things:

The "validate_flag" (func 4) is what interests us:

This function takes one parameter, sets up a local frame and delegates everything to func (;3;) returning it’s result (call 3 instruction)

Function 3 is the core validation point it:

  • Loads the flag from memory
  • Loads an IV from memory offset 66080
  • Prepares a local buffer
  • Calls func 2 parsing:
    • Output buffer
    • IV pointer
    • Ciphertext pointer (data[0] at offset 65536)

Function 2 is the decryption wrapper It takes 4 arguments :

  • Output buffer
  • IV pointer
  • Ciphertext pointer
  • Temp memory

And :

  • Sets up memory frames
  • Copies the IV into local space
  • Calls func 6 to perform decryption

Function 6 sets up memory and calls func 5

Function 5 :

  • Expands an AES key
  • Copies data from memory offset 66096
  • Reads the ciphertext from memory (offset 65536)
